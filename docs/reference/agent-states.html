Agent states

What each agent-status glyph means, and when it appears — per coding tool and in the terminal at large.

An agent state is what a coding agent (Claude Code, Codex, OpenCode, Pi) reports about itself while it runs inside an awesoMux pane. awesoMux resolves that into the glyph you see on a sidebar row, peek card, or status dot. The model has three layers:

1 — Execution state
What the agent is actually doing right now: idle, running, waiting, thinking, output, done, error. The resting truth.
2 — Attention overlay
An attentionReason (permission, approval, blocking choice, bell…) layered over the resting state. When set, the tile shows Needs input until you acknowledge. Normal turn-end is not an overlay.
3 — Display glyph
The single badge you see — the louder of overlay vs. execution state, drawn with a distinct shape and color. The legend below.

Glyphs and colors below are drawn to match the app exactly — both Catppuccin Mocha (dark) and Latte (light) themes, with shapes traced from AgentTile.swift. The shape (not just the hue) carries the distinction so the states stay legible without color.

The glyph legend

Eight display states, ordered by priority (lower number = louder; the loudest pane wins the sidebar badge).

Needs inputpriority 1 · loudest
Blocked on an explicit decision: permission, approval, or a provider choice. Peach badge with !. It is an overlay — acknowledge it and the resting execution state shows through. Normal turn completion can still increment unread / notify, but it stays visually Waiting.
Errorpriority 2
A live detector or runtime path saw an error while the shell is still running, or a restored snapshot carried an unresolved error. Red badge with x. Clears to idle when the next shell command in that pane exits zero.
Thinkingpriority 3
A tool call or model response is in flight. Mauve spinner arc (animated in the app). This is where the agent spends most of an active turn.
Donepriority 4
An agent run or detected agent command completed successfully. Teal checkmark. Often ephemeral — a process exit may close the pane before you see it. A terminal process-exit workspace close does not set this.
Outputpriority 5
Fresh stdout or visible progress in the recent-output window. Green dot. A provider only earns this if it exposes reliable progress — it is never faked from generic tool-end hooks.
Waitingpriority 6
Agent is alive but awaiting your next turn; no work in flight. Blue badge with a pause glyph (two vertical bars — deliberately not a caret or block-cursor, so it stays distinct from Running's triangle for color-blind users and does not read as a blinking text cursor). Normal turn-end lands here immediately; if it happened unfocused, unread / notification is added without changing the glyph to !. Reached only via an explicit runtime signal, never inferred from screen text.
Runningpriority 7
Agent attached with no recent activity — not raw shell-process liveness. Sapphire play triangle. The catch-all "alive, idle-ish" state before a more specific signal arrives.
Idlepriority 8 · quietest
No agent attached, or a shell-only session at rest. A faint dot on a neutral badge. Also the resting state for a fresh, never-prompted session and the target of a clean session-exit reset.

Each pair shows Mocha (left) and Latte (right). In the collapsed sidebar rail, most states render as a bare colored dot; only Needs input and Error keep their glyph, and Idle shows no badge at all.

The attention overlay

Needs input is not a resting state — it is an overlay. When any attentionReason is set, the tile projects to Needs input regardless of what the agent is resting on underneath. Acknowledging (selection dwell, terminal input, or "Mark as Read") clears the overlay and reveals the execution state beneath.

resting waiting + attentionReason → shows Needs input acknowledge → back to waiting

The six attention reasons:

permissionPrompt — blocked on approval userInputRequired — explicit question / choice bell — terminal bell desktopNotification processError unknown

Lifecycle

The target state contract. Attaching an agent moves a pane out of idle; work flows through thinking / output; normal turn-end rests on waiting; blocking choices raise the needsAttention overlay.

Solid edges are forward transitions; dashed edges are recovery / acknowledgement. The peach needsAttention overlay can rise from any active state and settles back to the resting state underneath on acknowledgement.

When they appear, per tool

Each agent emits its own hooks/events; awesoMux maps them to an execution state, an optional attention reason, and a runtime phase. The shared rows are identical across providers; the deltas are called out per tool.

A turn-end is blue pause immediately: Stop maps to waiting with no attention overlay. If it completes unfocused, awesoMux still adds unread / notification treatment. The PermissionRequest hook is load-bearing — Claude Code does not always populate the notification subtype (claude-code#11964), so the notification path is a best-effort fallback only.

OpenCode · opt-in

A plugin translates OpenCode surfaces into synthetic hooks, then the shared mapping applies. Adds StopFailure and a flat Notification:

OpenCode surface Synthetic hook Result
session.createdSessionStartidle
chat.messageUserPromptSubmitthinking
tool.execute.beforePreToolUsethinking
tool.execute.afterPostToolUsethinking
permission.askPermissionRequestneedsAttention (permissionPrompt)
session.idleStopwaiting (unfocused: unread / notify)
session.errorStopFailureerror

session.status is deliberately not used as a turn-start signal — it is a state snapshot, so chat.message carries the real UserPromptSubmit. OpenCode has no normal quit hook, so its glyph may persist after the agent process exits inside a living shell.

Pi · opt-in

A Pi extension maps native events to synthetic hooks:

Pi event Synthetic hook Result
session_startSessionStartidle
before_agent_startUserPromptSubmitthinking
tool_execution_startPreToolUsethinking
tool_execution_endPostToolUsethinking
agent_endStopwaiting (unfocused: unread / notify)
session_shutdownSessionEndshell idle (full reset)

Pi has no current permission template — a future permission hook should map to needsAttention (permissionPrompt). Of the four providers, Pi is the only one with a reliable session-exit signal (session_shutdown), so it cleanly resets the tile back to shell idle.

In the terminal, in general

Not every signal comes from an agent. Three terminal-level mechanisms feed the same status surface.

Shell activity
Separate from agent state. From libghostty prompt markers: idle (at prompt) or busy (running a command). A busy shell shows the Running label without touching AgentState. Debounced 250ms, so quick commands (ls, cd, git status) stay Idle — it signals "something big is running," not every command.
Bell & notifications
A terminal bell maps to the bell attention reason; a desktop-notification trigger to desktopNotification. Both raise the Needs-input overlay. Note: OSC desktop notifications are not parsed as agent runtime events — arbitrary terminal output could forge them.
Runtime side channel
The preferred transport — protocol awesomux-agent-v1. Each event is one JSONL line written to a per-pane file; visible-text polling is only a fallback for older/unconfigured panes and may never claim waiting.

awesoMux injects these per-pane environment variables so an in-pane process can emit events:

AWESOMUX_AGENT_EVENT_PROTOCOLProtocol name (awesomux-agent-v1)
AWESOMUX_SESSION_IDawesoMux session UUID
AWESOMUX_PANE_IDawesoMux pane UUID
AWESOMUX_AGENT_EVENT_FILEJSONL event file for this pane
AWESOMUX_AGENT_HOOKAbsolute path to the bundled helper
AWESOMUX_AGENT_ENABLED_SOURCESComma-separated enabled OpenCode/Pi sources

A runtime event carries a phase — the lifecycle moment, independent of the resolved state:

sessionStart promptSubmit toolStart toolEnd notification stop sessionEnd rename
